## This machine

Windows host with Ubuntu via WSL2, NVIDIA GeForce RTX 5060 Ti, 16GB VRAM.

Do all the work from the Ubuntu/WSL side, not native Windows — the repo's
scripts, paths, and conventions all assume a Linux shell. WSL2 uses the
Windows NVIDIA driver directly; you do NOT need to install a separate Linux
GPU driver inside WSL. Before installing anything, confirm the GPU is
actually visible from inside WSL:

    nvidia-smi

If that fails, stop and tell the user — it usually means the Windows-side
NVIDIA driver needs updating (CUDA-on-WSL support requires a recent driver),
not something to work around from inside Linux.

16GB VRAM is comfortably enough for the 2B Qwen2-VL backbone in fp16/bf16
with room to spare, and should handle either 7B model (Qwen2-VL-7B or
LLaVA-1.5-7B) one at a time in fp16 — watch nvidia-smi for headroom, and
fall back to 8-bit (bitsandbytes) loading only if you actually hit an OOM.
Don't try to load both 7B models into VRAM simultaneously on one GPU — if
both people are working this machine at the same time, coordinate so only
one heavy model is resident at once.

Set up the environment inside WSL:

    cd <repo>
    python3 -m venv .venv && source .venv/bin/activate
    pip install -r requirements.txt   # installs both the lightweight and GPU sections
    python3 -c "import torch; print(torch.cuda.is_available(), torch.cuda.get_device_name(0))"

That last line must print `True` and the 5060 Ti's name before you proceed —
if it prints False, the torch build installed is CPU-only; reinstall torch
with a CUDA wheel matching your driver (see pytorch.org's install selector)
rather than working around it.

## CheXpert data + loader gap (shared infrastructure — whoever gets there first does it, the other pulls the result)

Get access: https://stanfordmlgroup.github.io/competitions/chexpert/ or
https://aimi.stanford.edu/datasets/chexpert-chest-x-rays — both require a
data-use-agreement/registration (unlike OpenI). Check with the professor
first — the lab may already have a copy on a shared drive/server, or you may
need to register yourselves. The smaller "CheXpert-small" release is fine to
start with; you don't need the full-resolution ~440GB version.

`src/data/dataset.py`'s `load_chexpert_dataset()` is currently a stub in two
important ways — fix both before running any ablation on CheXpert:

  1. It hardcodes `report_text=""` with a comment "CheXpert has no free-text
     report, only labels." Check whether your actual CheXpert copy includes
     raw report text files (some registered-access copies do, since the
     public labels were originally extracted from real reports via NLP). If
     you only have the label CSV, you'll need to build a pseudo-report
     string per study from the finding columns (e.g.
     `"Cardiomegaly: positive; Edema: negative; ..."`) so there's SOMETHING
     to feed the "text" modality — this label-derived text is exactly what
     the paper means by CheXpert being "shorter and more label-centric"
     (Section 4.1) compared to OpenI's free-text reports.
  2. It hardcodes `label=None`. Pick one target for the AUC classification
     task — the natural choice is CheXpert's "No Finding" column inverted
     (`label = 1 - No_Finding`, i.e. 1 = abnormal) to match the
     "0=normal/1=abnormal" convention the OpenI loader already uses.
     Decide how to handle CheXpert's uncertain label value (-1) — the
     standard approaches are "u-zeros" (treat -1 as 0/negative) or "u-ones"
     (treat -1 as 1/positive); pick one, and note which one you used in the
     run's `config_snapshot` so it's reproducible.

Also double-check the `Path` column format in the actual CSV you get lines up
with how `load_chexpert_dataset()` joins it to `images_root` — CheXpert's
released CSVs have used a couple of different path-prefix conventions across
releases.

## Shared results log — both tracks write to the SAME file

    from src.evaluation.results_store import new_run_record, append_run
    import config

    run_record = new_run_record(
        mode="ablation",  # or "hallucination", "agentic_reasoning", "single", etc.
        config_snapshot={
            "seed": config.RANDOM_SEED, "n_studies": ..., "dataset": "openi",
            "model_family": "qwen2-vl",  # or "llava-1.5-7b" -- ALWAYS set this
        },
        results={"ablation": {...}},  # only the keys this run actually produced
    )
    append_run(config.RESULTS_LOG_PATH, run_record)

Always set `model_family` in config_snapshot so the two tracks' runs are
distinguishable in the same log without colliding. Never write
results/experiment_log.json by hand or truncate it — append_run already
loads-and-appends atomically.

After each meaningful real-data run: `git pull origin main` first (to pick up
anything the other person logged), then commit results/experiment_log.json
plus any code changes, and `git push origin main`. If git reports a merge
conflict inside results/experiment_log.json, resolve it by keeping BOTH
sides' new entries in the "runs" array (it's append-only by design — a
conflict there means two runs happened concurrently, not that one side is
wrong). Never resolve by discarding the other person's entries.

Sanity-check your numbers against the "Expected results to sanity-check
against" section at the bottom of CLAUDE.md — same ballpark and direction,
not exact values, since your dataset subset/seed will differ. If numbers are
wildly off in shape or direction, treat that as a bug to investigate, not
something to force-fit.
